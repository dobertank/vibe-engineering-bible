# Библия вайб-кодинга

Опорная дихотомия Simon Willison: **vibe coding ≠ vibe engineering**. Вайб-кодить — нормально и иногда необходимо. Всё, что выходит за пределы личного эксперимента, проходит через инженерный фильтр.

В этой библии — два завета:

- **Завет Свободы** (T0, vibe coding) — короткий, разрешительный. Личный эксперимент без ритуалов в обмен на три нерушимые границы.
- **Завет Дисциплины** (T1+, vibe engineering) — длинный, строгий. Восемь заповедей, DoD по уровням, spec-driven workflow, чек-лист ревьюера, роли релиза.

Этот документ — фильтр между ними. Переход из первого во второй — через явные ворота (§5 DoD), не сползанием.

## 1. Зачем

Мы строим AI-платформу и сами — крупнейшие пользователи Claude Code, Cursor, Codex, Claude Cowork. Наши практики становятся витриной, наши ошибки — публичными. AI кратно ускоряет ввод кода, но независимая эмпирика 2024–2026 показывает обратную сторону:

- **METR (июль 2025, 16 опытных разработчиков, 246 задач):** с AI-инструментами завершают задачи на 19% медленнее, при субъективной уверенности в ускорении на 20%.
- **GitClear (211M строк, 2020–2024):** дублирование блоков выросло в 8 раз, churn 3,1% → 5,7%, доля рефакторинга упала.
- **Spracklen et al. (USENIX Security 2025):** 19,7% рекомендованных LLM пакетов не существуют (5,2% у коммерческих, 21,7% у open-source); 43% галлюцинированных имён повторяются во всех 10 прогонах — slopsquatting предсказуем.
- **Veracode (2025, 80 задач × 100+ моделей):** 45% AI-кода падает на OWASP-классе уязвимостей; Java — 70%.
- **Stanford / Perry et al. (CCS 2023):** разработчики с AI пишут менее безопасный код и при этом увереннее в его безопасности.
- **CodeRabbit (декабрь 2025, 470 PR):** 1,7× критических дефектов; security 1,5–2×; производительность 8×.
- **Apiiro (2025):** 4× скорость поставки, 10× security risks; +322% privilege escalation paths, +153% архитектурных дефектов.

Из этого вытекают два завета: Свободы для T0 и Дисциплины для всего, что вышло за личный эксперимент.

## 2. Завет Свободы (T0)

T0 — это инкубатор, не лазейка. Здесь живёт всё, что не вышло за пределы твоей машины и твоей головы: бенчмарки моделей, обучение, скрипты на час, разведка идеи, эксперимент с фреймворком.

**Что разрешено без ритуалов.** Любые языки за пределами paved road, любые модели, любые промпты. Не нужны: спецификация, чужое ревью, тесты, README, ADR, threat model, SAST, маркировка `[ai]`, sunset-дата, регистрация в каталоге.

**Три нерушимые границы.** Завет Свободы заканчивается там, где начинается одна из них:

1. **Никаких прод-данных и клиентских данных.** Только синтетика или анонимизированный сэмпл из публичных источников. PII в T0 — нарушение, не недосмотр.
2. **Никаких рабочих секретов.** Никаких корпоративных API-ключей, токенов, креденшелов БД, refresh-токенов на твою учётку SSO. Если эксперимент требует прод-ключа — оформляй как T1 и получай ключ через стандартный канал.
3. **Никакого shared state.** Не пишешь в общие БД, не публикуешь в общие очереди, не вызываешь продакшен-эндпоинты, не светишь в продовом AI-Gateway без выделенной квоты. T0 живёт на твоём ноутбуке, в личном sandbox или dev-инстансе.

**Срок жизни.** ≤30 дней — норма. >30 дней без перехода в T1 — это либо удаление, либо миграция. T0, незаметно проживший в шкафу полгода и внезапно понадобившийся коллеге — это T1, который не прошёл ворота. Возвращайся к §5 DoD и проходи.

**Переход в T1** запускается одним из событий: показал коллеге за пределами своей команды, поделился ссылкой, попросил включить в roadmap, начал использовать сам ежедневно. С этого момента — README, маркировка `PROTOTYPE`, sunset-дата, регистрация в каталоге, ревью изменений.

**Тон.** Свобода — не привилегия и не уступка, а инвестиция: чем дешевле T0, тем больше идей доходит до проверки. Завет Дисциплины не существует, чтобы наказать за вайб; он существует, чтобы вайб не утащил продакшен.

## 3. Завет Дисциплины (T1+)

Всё, что вышло из T0, живёт по Завету Дисциплины. Здесь восемь заповедей, DoD по уровням, spec-driven workflow, чек-лист ревьюера AI-кода, baseline для CLAUDE.md/AGENTS.md, план внедрения и роли релиза. Не идеология — операционные правила. Чем дальше зона от T0 (T1 → T2 → T3), тем плотнее их сетка.

Заповеди задают общую модель. DoD приземляет её по зонам. Spec-driven — договор о требованиях до кода. Чек-лист — момент истины на ревью. Baseline — то, что AI-агент видит и слышит каждый запуск. Внедрение — как мы движем всё это через 2000+ человек, не превращая в карго-культ.

## 4. Восемь заповедей

### I. Кто закоммитил — тот и владеет
Пишет AI — отвечаешь ты. Защиты «это сгенерировал Claude» не существует. Правило Willison: **не коммитить код, который не сможешь подробно объяснить другому человеку**. Git blame — это контракт: твоё имя означает «прочёл, понял, протестировал». Tag `[ai]` информирует ревьюеров, но не освобождает.

### II. Прототип и прод — два разных кода
Каждая строка живёт в одной из четырёх зон: **T0** (личный эксперимент, ≤30 дней), **T1** (прототип, sunset-дата обязательна), **T2** (внутренний инструмент), **T3** (продакшен). Переход между зонами — только через явные ворота (см. §5 DoD). Прототип, проживший >30 дней, регистрируется в каталоге с owner-командой и sunset-датой; после 90 дней без коммитов — `deprecated`, после 180 — `archived`. Никаких «временно в проде»: временное живёт дольше всего.

### III. Демо — это контракт
Что показал — то и обещал. Защиты обязательны и кумулятивны: (1) визуальная маркировка `PROTOTYPE — NOT PRODUCTION`, баннер, `[DEMO]` в title; (2) **framing slide** перед демо: что реально / что Wizard-of-Oz / чего нет / сколько до прода; (3) email-резюме с теми же 4 пунктами. **Правило 10–30×:** время до production-grade = время демо × 10–30. Никакая дата запуска не объявляется на основе демо без согласования с релиз-менеджментом.

### IV. Один стек, один paved road
Прод-языки — **Python, JavaScript, Go, C++**. Точка. Новый язык — через ADR и одобрение Principal Engineer Community. Каждый новый язык в проде — новая on-call ротация, SAST-правила, CI-пайплайн, knowledge gaps. Сходить с paved road можно — но без бесплатной поддержки платформенной команды и с обоснованием trade-offs в ADR.

### V. Зависимость без аудита — атака
Каждый новый импорт от AI проверяется на существование, репутацию и совпадение имени до мёрджа: Socket.dev / Snyk / Dependabot. CI-проверка против базы известных hallucinated package names. Пакеты моложе 30 дней с <1000 загрузок вне нашей организации — ручной approval security-команды. **Любая новая runtime-зависимость требует обоснования в PR-описании, пиннинга версии и обновления lockfile в одном коммите.** Локальные MCP-серверы и сторонние Cursor-расширения — только из allow-list (CVE-2025-54135, CVE-2025-59944 — доказанные RCE через MCP в 2025).

### VI. Lethal trifecta не пересекается
Формулировка Willison: **(приватные данные) + (недоверенный контент) + (внешний канал)** одновременно у одного агента — гарантированный exfiltration. Минимум одну стенку держим. Промптовых фильтров недостаточно — это архитектура, а не текст.

Конкретно:
- AI не получает доступа к продакшен-секретам никогда. Никаких `.env` в контексте.
- Внешние чатботы (ChatGPT, Gemini, Perplexity) запрещены для непубличных данных, кода, секретов, клиентских данных.
- `--dangerously-skip-permissions` — только в изолированных sandbox-окружениях.
- Любое необратимое действие агента (запись в БД, push, payment, email) — через явный human approval (12-factor agents, фактор 7).

### VII. Ревью AI-кода — другой жанр
Не опечатки ищешь, а: галлюцинированные API, тавтологические тесты, удалённые ассерты, чрезмерные абстракции, скрытое расширение скоупа. **PR-size:** цель ≤200 LOC, потолок 400 LOC — больше дробится. **Auto-merge запрещён** для AI-помеченных PR во всех зонах выше T0. **T3 требует двух ревьюеров**, один — senior+. **Mutation testing** для критичных модулей: AI любит писать «всегда зелёные» тесты. Тесты пишутся отдельным запросом или другим человеком (Meta JiTTests: 4× рост ловимых дефектов при независимой генерации).

### VIII. Тесты, документация, наблюдаемость — не «потом»
Без них это файл с инструкциями для будущего археолога. Минимум для всего, что проживёт >1 дня:

- **README** на одну страницу: что, как запустить, кто owner, lifecycle.
- **CLAUDE.md / AGENTS.md** в корне репо (см. §8) с командами build/test/lint и «do not touch» путями.
- **Хотя бы один тест** на каждый изменённый юнит. Покрытие — не цель; fault detection — цель.
- **Структурированные логи в stdout**, не в файлы. Для агентов — audit trail на prompt/tool-call/result.
- **ADR** для каждого нетривиального архитектурного решения уровня T2+.
- **Живой spec в `openspec/`** для T2+: что система должна делать — в `specs/<capability>/`, что меняется в текущем PR — в `changes/<x>/`. Чат — контекст, спека — контракт. Подробнее — §6.

## 6. Spec-driven workflow

Для нетривиальных изменений T2+ договариваемся о спецификации **до** кода. Чат — контекст, спека — контракт. Live spec живёт в репо, а не в Confluence.

**Когда обязателен proposal.**

- T2+: новые capability, изменения API/контрактов, изменения модели данных, изменения security-границ. Для T3 — всегда, до открытия релизного тикета.
- Не нужен: T0/T1, баг-фиксы и рефакторинги без изменения внешнего поведения — сразу PR.
- Hot-fix: proposal оформляется пост-фактум, не позднее 24 часов после вывода в ПРОМ.

**Артефакты в `openspec/`.**

```
openspec/
├── specs/<capability>/spec.md         — живой спек (источник истины)
└── changes/<kebab-name>/
    ├── proposal.md                    — Why / What Changes / Impact
    ├── specs/<capability>/spec.md     — delta: ADDED / MODIFIED / REMOVED
    ├── design.md                      — как реализуем (нетривиальные решения)
    └── tasks.md                       — нумерованный чек-лист реализации
```

**Формат requirements.** SHALL/MUST в `### Requirement`; сценарии — GIVEN/WHEN/THEN. Формулировки в продуктовом языке, не реализационном.

**Lifecycle.** Proposed → Approved (PR на proposal зелёный) → Implemented (галочки в `tasks.md` закрыты, код смёрджен) → Archived (`changes/archive/YYYY-MM-DD-<x>/`, delta смёрджена в `openspec/specs/`).

**Связь с релизом T3.** Proposal согласован до выпуска дистрибутива; ИФТ/НТ проверяют соответствие спеку, а не «что получилось»; archive — после вывода в ПРОМ.

**Paved road.** Fission-AI OpenSpec (`npm install -g @fission-ai/openspec`, `openspec init`). Slash-команды `/opsx:propose`, `/opsx:apply`, `/opsx:archive` доступны в Claude Code, Codex, Cursor (~30 агентов). Сходить с paved road можно через ADR; артефакты proposal/specs/tasks обязательны независимо от инструмента.

## 7. Чек-лист ревьюера AI-кода

Печать и на стену рядом с каждым ревьюером. Семь вопросов, на каждый — ответ «да» или PR на доработку.

1. **Скоуп.** Меняет ли PR только то, что было запрошено? Diff конфинирован к плану — никаких «улучшений» соседних файлов.
2. **Галлюцинации API.** Каждый новый импорт, метод, флаг существует в установленной версии библиотеки?
3. **Тесты.** Тестируют ли тесты требование, а не реализацию (не тавтологии)? Не удалены ли упавшие тесты вместо починки? Mutation testing для критичного кода проходит?
4. **Безопасность.** Параметризованные SQL/ORM (никаких f-string SQL). Input validation на границах доверия. AuthZ на изменённых эндпоинтах. Никаких хардкоженных ключей. Output encoding для HTML. Никаких fail-open путей.
5. **Edge cases.** Empty / null / single element / unicode / большой ввод / network failure / DB-down — что происходит?
6. **Архитектура и зависимости.** Не дублирует существующую утилиту? Нет over-engineering. Нет deprecated API. Новые зависимости обоснованы, запинены, lockfile обновлён.
7. **Lethal trifecta** (для агентских PR): какие из трёх стенок (приватные данные / недоверенный вход / внешний канал) удержаны?
8. **Spec/proposal** (для T2+): соответствует ли PR утверждённому `proposal.md`? `tasks.md` закрыт галочками? Дельта в `openspec/changes/<x>/specs/` отражает реальное изменение требований, а не «что в коде получилось»? Расхождения зафиксированы в `design.md` или новой итерации proposal?

**Финальный вопрос ревьюеру.** Можешь ли ты объяснить каждую строку этого PR кому-то ещё? Если нет — отправляй на доработку.

## 8. CLAUDE.md / AGENTS.md baseline

Корпоративный baseline-шаблон лежит в **`CLAUDE.md.template.md`** и является единственным источником истины. Команды копируют его в корень своего репо как `CLAUDE.md` (для Claude Code) и/или `AGENTS.md` (для Codex, Cursor и других, поддерживающих agents.md). Корпоративную часть (разделы 1–7 шаблона) команды не редактируют — добавляют только репо-специфику в раздел 8.

**Принципы файла** (применяй при любых правках baseline):

1. **Краткость важнее полноты.** Цель — ≤2000 токенов. Anthropic: «раздутый CLAUDE.md заставит Claude игнорировать ваши инструкции». Если строку можно убрать без потери смысла — убери.
2. **Императив, а не описание.** «Используй pytest, не unittest» — да. «Этот проект исторически использует pytest» — нет.
3. **Конкретика, а не идеология.** Правила должны быть проверяемыми.
4. **Иерархия.** Корневой файл + локальные `CLAUDE.md` в подкаталогах для подсистем. Claude Code и Codex автоматически подгружают ближайший по дереву.
5. **Эмфаза работает.** «IMPORTANT» и «YOU MUST» измеримо повышают compliance моделей.

**Что НЕ класть:** длинные туториалы, дублирование линт-конфигов, маркетинг и историю проекта, секреты, универсальные best practices.

**Поддержка:** команда `#` в Claude Code добавляет инструкцию из чата прямо в CLAUDE.md — пользуйтесь каждый раз, когда корректируете агента. AI-Code Guild пересматривает корпоративную часть baseline раз в квартал.

## 9. Внедрение и роли

**Внедрение через paved road, не через мандат.** Большие изменения в инженерной культуре работают, когда правильный путь — самый лёгкий (Spotify Golden Paths, Netflix paved road). Жёсткие гейты без paved road загоняют команды в большие батчи и увеличивают blast radius на изменение (Charity Majors).

**Фаза 0 (0–4 недели).** Корпоративный AI-Gateway с логированием и блокировкой утечек. Allow-list инструментов: Claude Code, Cowork, Codex, Cursor. Зафиксировать T0–T3 и label `[ai]` в PR-template. Эталонный `CLAUDE.md.template.md` для четырёх языков.

**Фаза 1 (1–3 месяца).** Backstage-каталог с обязательными метаданными. Scaffolder для предзаполненных сервисов (CI, CODEOWNERS, AGENTS.md, наблюдаемость, secrets-vault). Базовые CI-laws: secret scan, license scan, slopsquatting-check, PR-size 400. AI-Code Guild с явной хартией.

**Фаза 2 (3–6 месяцев).** Soundcheck-метрики: % сервисов в каталоге, % с актуальной AGENTS.md, % AI-PR с маркировкой, % T3 с PRR. Сначала advisory, затем hard gate при ≥95%. Champions-программа: один senior+ на Tribe.

**Фаза 3 (6–12 месяцев).** Ежеквартальные DORA-замеры (deploy frequency, lead time, change failure rate, MTTR) — отдельно для AI-помеченных PR. Sunset-кампании по зомби-прототипам.

**Роли в жизненном цикле кода:**

| Роль | Что делает |
|---|---|
| Product Owner | Настройка ролей в Jira; заключение по выпуску; валидация аналитики с CISO; lifecycle-метаданные сервиса. |
| Delivery Manager | Драйвер релиза T3: подготовка сборки, релизный тикет, формирование PoB (если команда сама эксплуатирует), инициирование тестов и согласований, вывод в ПРОМ. |
| QA | ИФТ (всегда обязательна для T3). |
| Ответственный за НТ | Нагрузочное тестирование. |
| ЦКБ | Заключение по выпуску. |
| Проверки безопасности | SAST, lethal-trifecta аудит, threat model, slopsquatting-check; CISO-валидация аналитики. |
| Эксперт сопровождения | Заключение по выпуску; PoB при централизованном сопровождении; runbook; on-call; пейджер. |
| AI-Code Guild | Владеет этим документом и `CLAUDE.md.template.md`; пересмотр раз в квартал; чек-лист ревьюера; политики. |

**Главный KPI** — не процент соблюдения правил, а: (1) скорость от идеи до зарегистрированного прототипа (цель — минуты, не дни) и (2) доля прототипов, успешно прошедших T1 → T3 в течение года. Здоровое прототипное кладбище — то, где смерть происходит явно и быстро.

## 10. Глоссарий

**T0 / T1 / T2 / T3** — четыре зоны кода: эксперимент / прототип / внутренний инструмент / продакшен. Каждой зоне — свой DoD (см. §5).

**Vibe coding.** Karpathy (февраль 2025): «отдаться вайбам, забыть, что код существует». Подходит для T0.

**Vibe engineering.** Willison: AI-ассистированная разработка с инженерной дисциплиной. Наш дефолт для всего, что выходит за T0.

**Hallucination.** Уверенный вывод модели, не соответствующий реальности: несуществующий метод, выдуманный ключ конфига, неработающий импорт.

**70% problem.** Osmani: AI делает 70% видимой работы за вечер; оставшиеся 30% (edge cases, безопасность, перформанс, реальные данные) занимают в разы больше и определяют, доедет ли продукт до прода.

**Lethal trifecta.** Willison: одновременное наличие у агента (1) приватных данных, (2) недоверенного входа, (3) внешнего канала. Промпт-фильтры не работают — нужна архитектурная стенка.

**Slopsquatting.** Атакующий регистрирует пакет с именем, которое часто галлюцинирует LLM. RCE у разработчика, скопировавшего AI-вывод. 19,7% галлюцинированных имён, 43% повторяемы.

**Prompt injection.** Атака, при которой недоверенный текст в данных (issue, файл, email) переопределяет инструкции агента.

**Paved road / Golden Path.** Поддерживаемый путь по умолчанию (Spotify, Netflix). Сходить можно — через ADR, без бесплатной поддержки платформы.

**AGENTS.md / CLAUDE.md.** Файлы в корне репо с инструкциями для AI-агентов: команды, code style, «do not touch». См. §8 и `CLAUDE.md.template.md`.

**MCP (Model Context Protocol).** Открытый протокол Anthropic для подключения инструментов к LLM. Стал стандартом, но MCP-серверы — частый вектор атаки.

**Eval.** Набор `(input, expected output)` для оценки AI-системы. Без evals улучшение — слепое.

**AI-Gateway.** Централизованный шлюз ко всем LLM-провайдерам с логированием, фильтрацией секретов, квотами, атрибуцией расходов.

**ADR (Architecture Decision Record).** Шаблон Nygard: 1 страница, секции Context / Decision / Consequences. Immutable; изменение — новый ADR со ссылкой на старый.

**Spec-driven development (SDD).** Договорились о спецификации до кода. Чат — контекст, спека — контракт. См. §6.

**Delta spec.** Изменение требований в формате `ADDED / MODIFIED / REMOVED Requirements` со сценариями GIVEN/WHEN/THEN. Живёт в `openspec/changes/<x>/specs/`. После архивации мёрджится в `openspec/specs/`.

**OpenSpec.** Open-source CLI Fission-AI для SDD: `openspec init`, `/opsx:propose`, `/opsx:apply`, `/opsx:archive`. Поддерживает ~30 AI-агентов (Claude Code, Codex, Cursor и т.д.).

**Capability.** Логическая способность системы; имя директории в `openspec/specs/<capability>/`. Capability ≠ модуль кода: одна capability может пересекать несколько модулей и/или сервисов.

**SAST / DAST.** Static / Dynamic Application Security Testing. SAST — Semgrep, CodeQL. DAST — OWASP ZAP, Burp.

**Mutation testing.** Тестирование тестов: вносим мутацию (`==` → `!=`), смотрим, ловят ли. Зелёные после мутации — не тестируют.

**SLI / SLO / SLA.** Indicator (метрика) / Objective (цель) / Agreement (договорённость с клиентом).

**Canary deployment.** Постепенный rollout (1% → 10% → 50% → 100%) с авто-rollback по error-budget.

**Kill switch.** Feature flag для мгновенного отключения проблемной функциональности.

**Blast radius.** Масштаб ущерба от инцидента. Принцип «small blast radius» — ограничить распространение поломки.

**Fail-closed.** При отказе компонент закрывает доступ (безопасно). Для security всегда выбираем fail-closed.

**DORA.** Deployment Frequency, Lead Time, Change Failure Rate, MTTR — четыре ключевые метрики DevOps.

**12-factor agents.** Адаптация 12-factor app под LLM-агенты (HumanLayer): own your prompts, own your context, agent as stateless reducer, contact humans with tool calls.

**ИФТ / НТ.** Интеграционно-функциональное / нагрузочное тестирование. ИФТ — всегда обязательна для T3, НТ — за исключением hot-fix с согласованием сопровождения.

**PoB (Plan of Backout).** План отката релиза.
