# Дизайн: план внедрения «Библии вайб-кодинга»

**Дата:** 2026-05-10
**Автор брейнсторминга:** Aleksandr Prudchenko + Claude
**Целевой артефакт:** `docs/adoption-roadmap.md`

## 1. Цель документа

`docs/adoption-roadmap.md` — универсальный roadmap внедрения подхода из «Библии вайб-кодинга» для трёх размеров организаций (малая 1–10, средняя 10–100, крупная 100+). Документ заменяет Appendix B библии (раскрывая его и добавляя профили, которых раньше не было).

## 2. Аудитория и формат

- **Аудитория:** три профиля. Команда проходит decision tree, читает только свой раздел.
- **Глубина:** roadmap верхнего уровня (1–2 страницы на профиль). Без шаблонов кода, скриптов bootstrap, конкретных CI-config-сниппетов — это материал отдельного runbook-а, не этого документа.
- **Структурный подход:** «параллельные треки» — три независимых раздела по профилям + общая шапка и подвал. Альтернативы (концентрические круги по фазам, гибрид с общей Phase 0) рассмотрены и отклонены: приоритет — навигация при первом чтении.
- **Объём:** ~10–15 страниц всего.

## 3. Связь с библией

Новый документ **поглощает Appendix B библии**:
- При коммите `docs/adoption-roadmap.md` Appendix B удаляется из `vibe-engineering-bible.md`.

Что не дублируется (живёт в библии, ссылка отсюда):

## 5. Содержательные требования

### §1. TL;DR и как читать

- Внедрение библии = переключение команды из режима «AI используют, но никто не знает кто, как и с какими гарантиями» в режим «AI используют, и это видно в git, в каталоге, в DoD, в ревью».
- Документ построен по профилям. Пройди decision tree в §4, читай только свой раздел; склеивать best of из трёх — антипаттерн внедрения (см. §8).

### §2. KPI внедрения

Главный KPI — в §9 библии (время от идеи до зарегистрированного прототипа; доля прототипов, прошедших T1 → T3 за год). Здесь не дублируем — здесь добавляем специфику замера:

- **Baseline до старта.** Зафиксировать текущие значения обоих KPI в первую неделю Фазы 1 каждого профиля. Без baseline сдвиг не виден.
- **Ожидаемое время до сдвига.** Малая — 1–2 месяца. Средняя — 3–6 месяцев. Крупная — 6–12 месяцев. Раньше — шум, не сигнал.
- **Compliance-проценты** (% сервисов с CLAUDE.md, % AI-PR с тегом `[ai]`, % T3 с PRR) — вторичные. Пока главные KPI не растут, рост compliance — ритуал, не результат.

### §3. Принципы внедрения

Шесть штук, каждый — одна строка с обоснованием:
- Paved road > мандат (Spotify Golden Paths, Charity Majors).
- Advisory → hard gate, никогда наоборот.
- Сначала фиксация терминов и ролей — потом инфраструктура — потом метрики.
- Внедряй на одном пилотном репо до раскатки на всю организацию.
- Champions ≠ эскалаторы; их задача — устранять трение, не штрафовать.
- Внедрение измеряется месяцами, не неделями. Кварталы — нормальный шаг для крупной.

### §4. Decision tree выбора профиля

| Признак | Профиль |
|---|---|
| Один человек или команда из 2–10, нет отдельной платформы/security | I. Малая |
| 10–100 человек, есть общий бэклог, общий CI, нет paved-road-команды как роли | II. Средняя |
| 100+ (особенно >1000), есть платформа, security-governance, формальный релизный процесс | III. Крупная |

В крайних случаях (быстрый рост 5→50 за полгода) — берёт средний профиль и поэтапно повышает.

### §5. Профиль I. Малая команда (1–10)

**5.1. Старт: предусловия и стартовое состояние**

Предусловия:
- Один Git-репозиторий или их небольшое количество (≤10).
- AI-инструменты уже используются (или внедряются с нуля).
- Нет отдельной платформенной/security-команды; решения принимает 1–2 человека.

Оценка стартового состояния — 30 минут:
- Сколько активных репо? В скольких есть CLAUDE.md / AGENTS.md?
- Есть ли репо без owner-а — потенциальные `#zombie-t0`?
- Используется ли тег `[ai]` хоть как-то?

Если оценка занимает >2 часов — это уже не малая команда, переходи в профиль II.

**5.2. Фаза 1. Bootstrap (1–2 недели)**

- **Цель.** Зафиксировать в каждом активном репо: CLAUDE.md (универсальный шаблон) + lifecycle-метку (T0/T1/T2/T3).
- **Что появляется.** `CLAUDE.md` в корне (копия `CLAUDE.md.template.md`, заполнены §1 контекст и §2 стек, §8 пусто); тег зоны в первой строке; sunset-дата в README для T1+; Conventional Commits.
- **Роли.** Owner; co-owner для команды 2–10.
- **Gate.** В каждом активном репо — CLAUDE.md + lifecycle-метка; 0 бесхозных репо.
- **KPI.** 100% активных репо имеют CLAUDE.md; 0 «бесхозных» репо в продакшене.
- **Риски.** (а) Шаблон поставлен формально, не используется → первый PR после bootstrap-а пишет в §8 минимум одну реальную инструкцию; (б) пропуск зоны T0 → явный обзор «что у меня сейчас на ноуте?» в первый день фазы.

**5.3. Фаза 2. Sustain (постоянно)**

- **Цель.** Sunset-дисциплина и обновление CLAUDE.md по мере накопления повторов.
- **Регулярные действия.** Раз в квартал — обзор репо (T0 → T1, T1 → archived); каждый раз, повторив инструкцию агенту 3+ раза — добавить в §8; раз в полгода — посмотреть Appendix D библии.
- **Роли.** Owner.
- **Gate.** Нет, фаза постоянна. Триггер выхода — рост до 10+ человек → переход в профиль II.
- **KPI.** 0 zombie-репо (T0/T1 без коммитов >180 дней — заархивированы); §8 в CLAUDE.md растёт.
- **Риски.** (а) Bus factor 1 → co-owner для 2–10 обязателен, для соло — записка-завещание; (б) drift от библии → квартальный обзор.

### §6. Профиль II. Средняя команда (10–100)

**6.1. Старт**

Предусловия:
- Общий бэклог, общий CI, общий список репо (10–100).
- Tech Lead / Principal Engineer как фактический owner общих стандартов; нет отдельной платформенной команды.
- Нет CISO, ЦКБ, регламентированного T3-релиза.

Оценка стартового состояния — 1–2 дня:
- Список всех активных репо, lifecycle-зона, owner.
- В скольких есть живой CLAUDE.md (правлен за 90 дней)?
- Какие AI-инструменты используются, кем, через какой канал?
- **AI-Gateway сейчас:** есть ли централизованная точка вызова LLM, или каждый ходит со своим ключом? Если последнее — оцени, какие непубличные данные уходили во внешние API за 30 дней.
- Инциденты `#test-del`, `#sandbox-bypass`, `#promptable` (даже мелкие) — фиксируй.

При >100 репо или признаках регламентированного релиза — переходи в профиль III.

**6.2. Фаза 1. Assessment (2–4 недели)**

- **Цель.** Зафиксировать ландшафт; выбрать пилот.
- **Что появляется.** Инвентарный список (репо → зона → owner → последний коммит → CLAUDE.md да/нет); отчёт «что мы делаем плохо сейчас» (1 страница, 5–10 пунктов из реальной практики, отдельно — наличие/отсутствие AI-Gateway и фактические утечки); пилот — один T2 репо средней сложности.
- **Роли.** Owner политики (Tech Lead/Principal); security-владелец (один человек, часто совмещение); 1–2 champion-а.
- **Gate.** Инвентарь — 100% активных репо; пилот выбран.
- **KPI.** 100% репо с зоной и owner; 0 бесхозных T2/T3.
- **Риски.** (а) Assessment растягивается на квартал → assessment ≠ рефакторинг, пишем «как есть»; (б) пилот выбран по принципу «самый запущенный» → выбирать показательный, не запущенный.

**6.3. Фаза 2. Paved road, AI-Gateway, единые ворота (1–3 месяца)**

- **Цель.** На пилоте отработать paved road, развернуть **minimal viable AI-Gateway**, раскатать на T2/T3. Все ворота — advisory.
- **Что появляется.**
  - **Minimal viable AI-Gateway** — централизованная точка вызова LLM-провайдеров с тремя обязательными функциями: (а) фильтрация секретов в исходящем промпте, (б) аудит-лог промптов/ответов, (в) единая точка для allow-list инструментов и моделей. Кандидаты: open-source self-hosted (LiteLLM, Portkey, Helicone) или SaaS внутри корп-аккаунта (Anthropic Console workspace, OpenAI Org workspace) с принудительным SSO. Развёртывание — 1–2 человеко-недели, не квартал.
  - `CLAUDE.md.template.md` (универсальный) — копируется в пилот; в §4.1 явно: «вызовы LLM — только через `<имя AI-Gateway>`; внешние ChatGPT/Gemini/Perplexity напрямую запрещены». Затем — во все T2/T3.
  - CI-laws (advisory): `[ai]`-tag в title (warn); `Assisted-by:`-trailer (warn); PR-size warning ≥400 LOC; slopsquatting-check; license scan.
  - PR-template с полями «AI-инструменты / что я лично проверил / как тестировал».
  - Список paved-road языков (2–3) в общем wiki.
  - Лёгкий каталог lifecycle (таблица в wiki / Notion-страница; не Backstage).
- **Роли.** Те же + champion-ы раскатывают.
- **Gate.** 100% T2/T3 имеют CLAUDE.md; CI-laws advisory; пилот закрыл ≥5 AI-PR по новым правилам; AI-Gateway работает; ≥80% корпоративных AI-запросов идут через него.
- **KPI.** ≥80% T2/T3 с актуальным CLAUDE.md; ≥50% AI-PR с тегом `[ai]` и trailer; % AI-запросов через Gateway.
- **Риски.** (а) Shadow-stacks → ADR-маршрут осмысленен только при наличии paved road; (б) CI-laws игнорируются → warn-тон; fail в Фазе 3; (в) AI-Gateway-проект разрастается до квартала → MVP-критерий — три обязательные функции; (г) Gateway внедряется как fence без paved road → должен быть быстрее или равен прямому доступу, latency overhead ≤200 мс.

**6.4. Фаза 3. Hardening и метрики (3–6 месяцев)**

- **Цель.** Advisory → hard gate. Замер главного KPI; spec-driven workflow для T2+.
- **Что появляется.** Hard gate в CI (PR-size ≥400 — fail; slopsquatting — fail; license scan — fail; missing `[ai]`-tag при детектированном AI-паттерне — warn, не fail); замер главного KPI (baseline + раз в 2 месяца); первый sunset-цикл (T0/T1 без коммитов >180 дней — archived); `openspec/`-каталог в первом T2-репо: первый proposal → archived → delta смёрджена; threat model basic для всех T2; SAST без high/critical в CI на T2+.
- **Роли.** Те же + security-владелец активнее.
- **Gate.** ≥95% T2/T3 с CLAUDE.md; ≥80% AI-PR с тегом; первый sunset-цикл прошёл; главный KPI измерен дважды; первый openspec-archive есть.
- **KPI.** Главный KPI: время от идеи до прототипа сократилось vs baseline (или явно зафиксирована недвижимость — тоже сигнал); доля zombie-репо снизилась.
- **Риски.** (а) Hard gate без paved road → большие батчи (Charity Majors); включается только при «≥80% AI-PR прошли advisory без жалоб»; (б) sunset-кампания превращается в «никто не удаляет» → критерий по последнему коммиту, не статусу; (в) spec-driven формально (proposal после кода) → первое archived изменение демонстрируется на ретро.

### §7. Профиль III. Крупная организация (100+, особенно >1000)

**7.1. Старт**

Предусловия:
- 100+ инженеров; особенно >1000.
- Платформенная команда (DX) как отдельная роль.
- Формальный security-governance: CISO; возможно ЦКБ.
- Регламентированный T3-релиз по Appendix A библии или прямой аналог.
- Executive sponsor для AI-инициативы (CTO / Chief Architect).

Оценка стартового состояния — 4 недели:
- Audit AI-инструментов: что официально, что shadow.
- Существующий каталог сервисов: Backstage / самописный / нет?
- Уже произошедшие инциденты с AI — фиксируй для Guild charter.
- Скан paved-road языков: что фактически в проде vs. задекларировано.
- Кандидаты в champions: 1 senior+ на каждый Tribe.

**7.2. Фаза 0. Mandate (0–4 недели)**

- **Цель.** Executive mandate, политика на бумаге, AI-Gateway корпоративный.
- **Что появляется.** Корпоративный AI-Gateway (логирование, фильтрация секретов, квоты, атрибуция расходов, SSO); allow-list AI-инструментов и MCP-серверов (с учётом CVE-2025-* из Appendix D); библия опубликована как корпоративный стандарт; PR-template организационного уровня; AI-Code Guild сформирована (5–10 человек, Principal+, явная хартия, owner библии и шаблонов).
- **Роли.** Executive sponsor (CTO/Chief Architect); Guild lead; Platform lead; Security lead (CISO или замещающий); Comms.
- **Gate.** AI-Gateway работает; ≥80% AI-трафика через него; Guild собралась минимум дважды и имеет хартию; шаблоны опубликованы; allow-list-ы зафиксированы.
- **KPI.** ≥80% AI-трафика через Gateway; allow-list зафиксирован.
- **Риски.** (а) Mandate без бюджета → executive sponsor + бюджет = gate выхода Фазы 0; (б) Gateway как fence без path → быстрее личного ChatGPT, иначе обходят; (в) allow-list расширительно «на всякий случай» → начинается с минимального, пополнение через Guild.

**7.3. Фаза 1. AI-Gateway корпоративный + paved road (1–3 месяца)**

- **Цель.** Backstage-каталог с метаданными; Scaffolder; CI-laws advisory.
- **Что появляется.** Backstage (или эквивалент) с обязательными метаданными (owner, lifecycle, T-зона, sunset); Scaffolder-шаблоны для новых сервисов (CI, CODEOWNERS, AGENTS.md, наблюдаемость, secrets-vault); CI-laws advisory; paved-road языки (3–5) с поддерживаемым стеком и on-call-ротациями; чек-лист ревьюера AI-кода как корпоративный артефакт.
- **Роли.** Те же + Guild ведёт ежемесячный «changelog политики» — 1 страница, всегда заканчивающаяся «что меняется в твоём CLAUDE.md».
- **Gate.** ≥80% сервисов в Backstage; Scaffolder выпускает ≥1 новый сервис в неделю; ≥5 пилотных Tribe-ов с CI-laws advisory.
- **KPI.** ≥80% сервисов в каталоге; ≥50% новых сервисов через Scaffolder; главный KPI измерен (baseline).
- **Риски.** (а) Backstage внедряется год → MVP-таблица сначала, миграция параллельно; (б) Scaffolder не отражает реальный стек → пилотный Tribe правит в первый месяц, валидация двумя сервисами вне пилота; (в) changelog в спам → 1 страница в месяц, обязательная секция «как это меняет твою работу».

**7.4. Фаза 2. Каталог + Soundcheck advisory (3–6 месяцев)**

- **Цель.** Soundcheck advisory; Champions; mutation testing на критичных модулях; spec-driven в paved-road-инструменте.
- **Что появляется.** Soundcheck advisory (% в каталоге, % с актуальной AGENTS.md, % AI-PR с тегом, % T3 с PRR); Champions-программа (1 senior+ на Tribe); mutation testing для критичных модулей T3 (mutmut/Stryker/PIT) — ночные прогоны, fail-blocking только на T3-релизе; threat model basic для T2, full для T3; spec-driven в paved-road-инструменте — ≥10 archived `openspec/changes/` к концу фазы.
- **Роли.** Те же + Champions активны на Tribe-уровне.
- **Gate.** Soundcheck — данные по 100% сервисов; ≥1 Champion на Tribe; ≥80% T2/T3 с CLAUDE.md, правленным за 90 дней; ≥10 archived openspec-changes.
- **KPI.** Главный KPI замерен повторно vs baseline; время от идеи до прототипа (индикативная цель -30…-50%); ≥80% Tribe-ов с активным Champion.
- **Риски.** (а) Soundcheck → hard gate раньше времени → gate только при ≥95% advisory без gaming, для этого — случайный аудит; (б) Champions становятся «AI police» → хартия явная, эскалация в Guild; (в) mutation testing замедляет CI → только критичные модули (≤10% кода), ночные прогоны.

**7.5. Фаза 3. Hard gates + DORA + sunset (6–12 месяцев)**

- **Цель.** Soundcheck advisory → hard gate; ежеквартальные DORA-замеры по AI-PR; sunset-кампании; устоявшийся ритм.
- **Что появляется.** Hard gate (PR не мержится без CLAUDE.md; без `[ai]`-tag при детектированном AI-паттерне — переход warn → fail происходит после стабильного adoption за ≥3 квартала подряд, иначе остаётся warn; без OpenSpec proposal для T2+ из скоупа §6 библии); ежеквартальные DORA отдельно для AI-PR vs human-only; квартальная sunset-«охота» (T0/T1 без коммитов >180 дней — archived автоматически); Guild-ритм (квартальный пересмотр библии по reasoned ADR; квартальные встречи Guild ↔ Champions).
- **Роли.** Устоявшиеся.
- **Gate.** Фаза постоянна; «выход» = установившийся квартальный ритм без авральных правок.
- **KPI.** Главный KPI стабильно низкий; DORA по AI-PR: change failure rate ≤ DORA-elite, lead time ≤ DORA-high; 0 zombie-репо в каталоге.
- **Риски.** (а) DORA gamed (теги только на безопасные) → квартальный случайный аудит на 5–10 PR с детектированием AI-паттерна; (б) hard gate без paved road → gate-критерий из Фазы 2 обязателен; (в) Guild «забалтывается» → ядро меняется только по reasoned ADR, ритм правок ядра не чаще раза в квартал.

### §8. Anti-patterns внедрения

Восемь именованных паттернов:

1. **Big-bang rollout** — пилот → раскатка, никогда наоборот.
2. **Hard gate с первого дня** — advisory → hard gate; gate включается при ≥95% advisory-compliance без gaming.
3. **Mandate без paved road** — paved road first; правила следствие, не причина.
4. **Compliance-метрика как главный KPI** — главные KPI всегда главные (см. §2).
5. **AI-Code Guild без executive sponsor** — Guild lead имеет прямой канал; квартальный sponsor-review.
6. **Champion как полицейский** — хартия явная: устранять трение, не штрафовать; эскалация в Guild.
7. **Склейка best of из всех профилей** — один профиль (см. §4), без подсматривания в соседние.
8. **Внедрение без baseline главного KPI** — baseline в первую неделю Фазы 1, не позже.

1. Покрывает все три профиля по единому шаблону фазы (цель → что появляется → роли → gate → KPI → риски).
