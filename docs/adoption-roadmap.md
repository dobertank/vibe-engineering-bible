# План внедрения «Библии вайб-кодинга»

> Универсальный roadmap внедрения подхода из «Библии вайб-кодинга» для трёх размеров организаций: малая (1–10), средняя (10–100), крупная (100+, особенно >1000). Документ строится по принципу параллельных треков: пройди decision tree в §4 и читай **только свой профиль**; склейка best of из всех трёх — антипаттерн (см. §8).

## 1. TL;DR и как читать

- Внедрение библии = переключение команды из режима «AI используют, но никто не знает кто, как и с какими гарантиями» в режим «AI используют, и это видно в git, в каталоге, в DoD, в ревью».
- Документ построен по профилям. Пройди decision tree в §4, читай только свой раздел; склеивать best of из трёх — антипаттерн внедрения (см. §8).

## 2. KPI внедрения

Главный KPI — в §9 библии (время от идеи до зарегистрированного прототипа; доля прототипов, прошедших T1 → T3 за год). Здесь не дублируем — здесь добавляем специфику замера:

- **Baseline до старта.** Зафиксировать текущие значения обоих KPI в первую неделю Фазы 1 каждого профиля. Без baseline сдвиг не виден.
- **Ожидаемое время до сдвига.** Малая — 1–2 месяца. Средняя — 3–6 месяцев. Крупная — 6–12 месяцев. Раньше — шум, не сигнал.
- **Compliance-проценты** (% сервисов с CLAUDE.md, % AI-PR с тегом `[ai]`, % T3 с PRR) — вторичные. Пока главные KPI не растут, рост compliance — ритуал, не результат.

## 3. Принципы внедрения

- Paved road > мандат (Spotify Golden Paths, Charity Majors).
- Advisory → hard gate, никогда наоборот.
- Сначала фиксация терминов и ролей — потом инфраструктура — потом метрики.
- Внедряй на одном пилотном репо до раскатки на всю организацию.
- Champions ≠ эскалаторы; их задача — устранять трение, не штрафовать.
- Внедрение измеряется месяцами, не неделями. Кварталы — нормальный шаг для крупной.

## 4. Как выбрать профиль

| Признак | Профиль |
|---|---|
| Один человек или команда из 2–10, нет отдельной платформы/security | I. Малая |
| 10–100 человек, есть общий бэклог, общий CI, нет paved-road-команды как роли | II. Средняя |
| 100+ (особенно >1000), есть платформа, security-governance, формальный релизный процесс | III. Крупная |

В крайних случаях (быстрый рост 5→50 за полгода) — берётся средний профиль и поэтапно повышается.

## 5. Профиль I. Малая команда (1–10)

### 5.1. Старт: предусловия и стартовое состояние

**Предусловия:**
- Один Git-репозиторий или их небольшое количество (≤10).
- AI-инструменты уже используются (или внедряются с нуля).
- Нет отдельной платформенной/security-команды; решения принимает 1–2 человека.

**Оценка стартового состояния — 30 минут:**
- Сколько активных репо? В скольких есть CLAUDE.md / AGENTS.md?
- Есть ли репо без owner-а — потенциальные `#zombie-t0`?
- Используется ли тег `[ai]` хоть как-то?

Если оценка занимает >2 часов — это уже не малая команда, переходи в профиль II.

### 5.2. Фаза 1. Bootstrap (1–2 недели)

- **Цель.** Зафиксировать в каждом активном репо: CLAUDE.md (универсальный шаблон) + lifecycle-метку (T0/T1/T2/T3).
- **Что появляется.** `CLAUDE.md` в корне (копия `CLAUDE.md.template.md`, заполнены §1 контекст и §2 стек, §8 пусто); тег зоны в первой строке; sunset-дата в README для T1+; Conventional Commits.
- **Роли.** Owner; co-owner для команды 2–10.
- **Gate-критерий выхода.** В каждом активном репо — CLAUDE.md + lifecycle-метка; 0 бесхозных репо.
- **KPI фазы.** 100% активных репо имеют CLAUDE.md; 0 «бесхозных» репо в продакшене.
- **Главные риски.**
  - *Шаблон поставлен формально, не используется.* Контрмера: первый PR после bootstrap-а пишет в §8 минимум одну реальную инструкцию из чата с агентом. Если через 2 недели §8 пуст — шаблон установлен, но не живёт.
  - *Пропуск зоны T0.* Личные эксперименты остаются без sunset-даты, превращаются в зомби. Контрмера: явный обзор «что у меня сейчас на ноуте?» в первый день фазы.

### 5.3. Фаза 2. Sustain (постоянно)

- **Цель.** Sunset-дисциплина и обновление CLAUDE.md по мере накопления повторов.
- **Регулярные действия.**
  - Раз в квартал — обзор репо: какие T0 проросли в T1? какие T1 умерли (>180 дней без коммитов = archived)? какие зоны нуждаются в перекатегоризации?
  - Каждый раз, повторив инструкцию агенту 3+ раза — добавить её в §8 CLAUDE.md.
  - Раз в полгода — посмотреть Appendix D библии: новые CVE/исследования, требующие правки §4 шаблона?
- **Роли.** Owner.
- **Gate-критерий выхода.** Нет, фаза постоянна. Триггер выхода из профиля — рост до 10+ человек, тогда переход в профиль II.
- **KPI фазы.**
  - 0 zombie-репо (T0/T1 без коммитов >180 дней — заархивированы).
  - §8 в CLAUDE.md растёт, не закостеневает.
- **Главные риски.**
  - *Bus factor 1.* Если единственный owner уходит в отпуск — внедрение замирает. Контрмера: для команды 2–10 — co-owner обязателен; для соло — короткая записка-завещание (где политика, как читать, что в §8).
  - *Drift от библии.* Появляются новые редакции — никто не пересматривает. Контрмера: квартальный обзор репо = квартальный пересмотр CLAUDE.md vs текущей библии.

## 6. Профиль II. Средняя команда (10–100)

### 6.1. Старт: предусловия и стартовое состояние

**Предусловия:**
- Общий бэклог, общий CI, общий список репо (10–100).
- Tech Lead / Principal Engineer как фактический owner общих стандартов; нет отдельной платформенной команды.
- Нет CISO, ЦКБ, регламентированного T3-релиза.

**Оценка стартового состояния — 1–2 дня:**
- Список всех активных репо, lifecycle-зона, owner.
- В скольких есть живой CLAUDE.md (правлен за 90 дней)?
- Какие AI-инструменты используются, кем, через какой канал?
- **AI-Gateway сейчас:** есть ли централизованная точка вызова LLM, или каждый ходит со своим ключом? Если последнее — оцени, какие непубличные данные уходили во внешние API за 30 дней.
- Инциденты `#test-del`, `#sandbox-bypass`, `#promptable` (даже мелкие) — фиксируй.

При >100 репо или признаках регламентированного релиза — переходи в профиль III.

### 6.2. Фаза 1. Assessment (2–4 недели)

- **Цель.** Зафиксировать ландшафт; выбрать пилот.
- **Что появляется.** Инвентарный список (репо → зона → owner → последний коммит → CLAUDE.md да/нет); отчёт «что мы делаем плохо сейчас» (1 страница, 5–10 пунктов из реальной практики, отдельно — наличие/отсутствие AI-Gateway и фактические утечки); пилот — один T2 репо средней сложности.
- **Роли.** Owner политики (Tech Lead/Principal); security-владелец (один человек, часто совмещение); 1–2 champion-а.
- **Gate-критерий выхода.** Инвентарь — 100% активных репо; пилот выбран.
- **KPI фазы.** 100% репо с зоной и owner; 0 бесхозных T2/T3.
- **Главные риски.**
  - *Assessment растягивается на квартал.* Симптом — попытка попутно почистить инвентарь, переименовать репо, мигрировать. Контрмера: assessment ≠ рефакторинг; пишем «как есть», не «как должно быть».
  - *Пилот выбран по принципу «самый запущенный».* Гарантирует провал. Контрмера: пилот выбирается по принципу «самый показательный для основного потока работы».

### 6.3. Фаза 2. Paved road, AI-Gateway, единые ворота (1–3 месяца)

- **Цель.** На пилоте отработать paved road, развернуть **minimal viable AI-Gateway**, потом раскатать на все T2/T3. Все ворота — advisory.
- **Что появляется.**
  - **Minimal viable AI-Gateway** — централизованная точка вызова LLM-провайдеров. Не полноценный enterprise-шлюз; цель — закрыть три риска: (а) фильтрация секретов в исходящем промпте, (б) аудит-лог промптов/ответов, (в) единая точка для allow-list инструментов и моделей. Кандидаты: open-source self-hosted (LiteLLM, Portkey, Helicone) или SaaS внутри корпоративного аккаунта (Anthropic Console workspace, OpenAI Org workspace) с принудительным SSO. Развёртывание — 1–2 человеко-недели, не квартал.
  - `CLAUDE.md.template.md` (универсальный) — копируется в пилот; в §4.1 явно фиксируется «вызовы LLM — только через `<имя AI-Gateway>`; внешние ChatGPT/Gemini/Perplexity напрямую запрещены». Затем — во все T2/T3.
  - CI-laws (advisory): `[ai]`-tag в title PR (warn); `Assisted-by:`-trailer в коммитах (warn); PR-size warning ≥400 LOC; slopsquatting-check на новые импорты (Socket.dev / Snyk / самописный против списка известных hallucinated names); license scan (FOSSA / Snyk).
  - PR-template с обязательными полями «AI-инструменты / что я лично проверил / как тестировал».
  - Список paved-road языков (2–3) опубликован в общем wiki.
  - Лёгкий каталог lifecycle — таблица в wiki или Notion-страница; не Backstage.
- **Роли.** Те же + champion-ы раскатывают по очереди.
- **Gate-критерий выхода.** 100% T2/T3 репо имеют CLAUDE.md; CI-laws включены advisory; пилот закрыл минимум 5 AI-PR по новым правилам (с тегом, с trailer, с заполненным PR-template); AI-Gateway работает; ≥80% корпоративных AI-запросов идут через него (детектируется по сетевым логам / отсутствию исходящих вызовов на api.openai.com / api.anthropic.com мимо Gateway).
- **KPI фазы.**
  - ≥80% T2/T3 с актуальным CLAUDE.md.
  - ≥50% AI-PR с тегом `[ai]` и trailer.
  - % AI-запросов через Gateway.
- **Главные риски.**
  - *Shadow-stacks.* Язык вне paved road используется тайно («я попробую Rust на этом микросервисе»). Контрмера: ADR-маршрут для схода с paved road работает, только если paved road реально есть; пока его нет — запреты бессмысленны, а с paved road — само собой выбирается.
  - *CI-laws игнорируются как «ещё одна штука».* Контрмера: warn в советующем тоне («твой PR 423 LOC — обычно ≤200; если уверен, продолжай»); fail включается в Фазе 3, не сейчас.
  - *AI-Gateway-проект разрастается до квартала.* Симптом — backlog с квотами, биллингом, dashboard-ами до того, как фильтр секретов работает. Контрмера: MVP-критерий — три обязательные функции (см. выше); всё остальное — после Фазы 3.
  - *AI-Gateway внедряется как fence без paved road.* Разработчики обходят через личные ключи, потому что Gateway медленнее. Контрмера: Gateway должен быть быстрее или равен прямому доступу; SSO-логин — один раз в день, не на каждый запрос; latency overhead ≤200 мс.

### 6.4. Фаза 3. Hardening и метрики (3–6 месяцев)

- **Цель.** Advisory → hard gate. Запустить замер главного KPI и spec-driven workflow для T2+.
- **Что появляется.**
  - Hard gate в CI: PR-size ≥400 LOC — fail; slopsquatting-check — fail; license scan — fail; missing `[ai]`-tag при детектированном AI-паттерне (>30% diff с типичными AI-сигнатурами) — warn, не fail (ложноположительные дороже).
  - Замер главного KPI: baseline зафиксирован в первую неделю, повторный замер каждые 2 месяца.
  - Sunset-кампания: первый цикл — все T0/T1 без коммитов >180 дней архивируются.
  - `openspec/`-каталог в первом T2-репо: первый proposal → archived → delta смёрджена в `openspec/specs/`.
  - Threat model basic для всех T2.
  - SAST без high/critical в CI на T2+.
- **Роли.** Те же + security-владелец активнее (threat model, SAST-разбор).
- **Gate-критерий выхода.** ≥95% T2/T3 с CLAUDE.md; ≥80% AI-PR с тегом; первый sunset-цикл прошёл; главный KPI измерен дважды (baseline + после Фазы 3); первый openspec-archive есть.
- **KPI фазы.**
  - Главный KPI: время от идеи до зарегистрированного прототипа сократилось vs baseline (или явно зафиксировано, что не сократилось — тоже сигнал).
  - Доля zombie-репо снизилась.
- **Главные риски.**
  - *Hard gate без paved road.* Команды загоняются в большие батчи и обходят (Charity Majors). Контрмера: hard gate включается только после KPI «≥80% AI-PR прошли advisory-стадию без жалоб».
  - *Sunset-кампания превращается в «никто ничего не удаляет».* Симптом — owner-ы переводят зомби в «поддерживается» без коммитов. Контрмера: sunset-критерий — последний коммит, не статус.
  - *Spec-driven workflow ставится формально.* `proposal.md` пишется после кода, не до. Контрмера: первое archived изменение демонстрируется на ретро как «вот как это работает»; потом champion-ы повторяют.

## 7. Профиль III. Крупная организация (100+, особенно >1000)

### 7.1. Старт: предусловия и стартовое состояние

**Предусловия:**
- 100+ инженеров; особенно актуально >1000.
- Существует платформенная команда (Developer Productivity / DX) как отдельная роль.
- Формальный security-governance: CISO; возможно ЦКБ.
- Регламентированный T3-релиз по Appendix A библии (ИФТ/НТ/PoB/4 заключения) или его прямой аналог.
- Есть executive sponsor для AI-инициативы (CTO / Chief Architect).

**Оценка стартового состояния — 4 недели:**
- Audit AI-инструментов: что используется официально, что shadow (личные ChatGPT-аккаунты).
- Существующий каталог сервисов: Backstage / самописный / нет?
- Уже произошедшие инциденты с AI (даже мелкие) — фиксируем для Guild charter.
- Скан paved-road языков: что фактически в проде vs. что задекларировано.
- Кандидаты в champions: 1 senior+ на каждый Tribe.

### 7.2. Фаза 0. Mandate (0–4 недели)

- **Цель.** Получить executive mandate, опубликовать политику, развернуть AI-Gateway как точку контроля и аналитики.
- **Что появляется.**
  - **Корпоративный AI-Gateway** — централизованный шлюз ко всем LLM-провайдерам с логированием, фильтрацией секретов, квотами, атрибуцией расходов, SSO.
  - **Allow-list AI-инструментов** (Claude Code, Cursor, Codex и т.д.) и **allow-list MCP-серверов** (см. CVE-2025-* в Appendix D библии — `CurXecute`, `mcp-remote`, `EscapeRoute`).
  - **Библия опубликована как корпоративный стандарт** (внутреннее издание); `CLAUDE.md.template.md` и `CLAUDE.md.template.enterprise.md` лежат в общем артефакт-репозитории.
  - **PR-template** организационного уровня с обязательными полями «AI-инструменты / что я лично проверил / как тестировал».
  - **AI-Code Guild сформирована**: 5–10 человек (Principal+), явная хартия, owner библии и шаблонов; ежеквартальный пересмотр.
- **Роли.**
  - **Executive sponsor** (CTO / Chief Architect) — mandate, бюджет.
  - **AI-Code Guild lead** (Principal+) — owner политики.
  - **Platform lead** — AI-Gateway.
  - **Security lead** (CISO или замещающий) — security-baseline.
  - **Comms** — коммуникация в организацию.
- **Gate-критерий выхода.** AI-Gateway работает; ≥80% корпоративных AI-запросов идят через него; Guild собралась минимум дважды и имеет хартию; шаблоны опубликованы; allow-list-ы зафиксированы.
- **KPI фазы.**
  - ≥80% AI-трафика — через AI-Gateway (детектируется по сетевым логам).
  - Allow-list зафиксирован, открытый процесс пополнения.
- **Главные риски.**
  - *Mandate без бюджета.* Guild — pet project; платформа не выделена. Контрмера: executive sponsor + бюджет — gate выхода Фазы 0.
  - *AI-Gateway внедряется как fence без paved road.* Разработчики обходят через личные ChatGPT-аккаунты. Контрмера: Gateway должен быть быстрее и удобнее личного ChatGPT; если медленнее — обход гарантирован.
  - *Allow-list заполняется расширительно «на всякий случай».* Внутри оказывается CVE-уязвимый MCP-сервер. Контрмера: список начинается с минимального; пополнение — только через Guild с обоснованием.

### 7.3. Фаза 1. AI-Gateway корпоративный + paved road (1–3 месяца)

- **Цель.** Развернуть каталог сервисов с метаданными, Scaffolder для предзаполненных сервисов, ввести CI-laws advisory.
- **Что появляется.**
  - **Backstage** (или эквивалент) с обязательными метаданными: owner, lifecycle, T-зона, sunset-дата.
  - **Scaffolder-шаблоны** для новых сервисов: CI, CODEOWNERS, AGENTS.md, наблюдаемость, secrets-vault.
  - **CI-laws (advisory)**: `[ai]`-tag в title, `Assisted-by:`-trailer, PR-size warning ≥400 LOC, slopsquatting-check, secret scan, license scan.
  - **Paved-road языки**: 3–5 штук, для каждого — поддерживаемый стек в Scaffolder, on-call-ротация, SAST-правила.
  - **Guild публикует чек-лист ревьюера AI-кода (§7 библии)** как корпоративный артефакт; champions раскатывают.
- **Роли.** Те же + Guild ведёт ежемесячный «changelog политики» — 1 страница, всегда заканчивающаяся «что меняется в твоём CLAUDE.md».
- **Gate-критерий выхода.** Backstage содержит ≥80% сервисов; Scaffolder выпускает ≥1 новый сервис в неделю; ≥5 пилотных Tribe-ов используют CI-laws advisory.
- **KPI фазы.**
  - ≥80% сервисов в каталоге с актуальными метаданными.
  - ≥50% новых сервисов создаются через Scaffolder (vs. ручной).
  - Главный KPI измерен (baseline до старта зафиксирован в Фазе 0).
- **Главные риски.**
  - *Backstage внедряется год вместо квартала.* Контрмера: MVP — простая таблица/Notion с теми же полями; миграция в Backstage — параллельно, не блокирует Фазу 1.
  - *Scaffolder-шаблон не отражает реальный стек.* Симптом — все его обходят. Контрмера: пилотный Tribe правит Scaffolder под себя в первый месяц; шаблон валидирован минимум двумя сервисами вне пилота до раскатки.
  - *Changelog политики превращается в спам.* Контрмера: 1 страница в месяц, не больше; обязательная секция «как это меняет твою работу прямо сейчас».

### 7.4. Фаза 2. Каталог + Soundcheck advisory (3–6 месяцев)

- **Цель.** Запустить Soundcheck-метрики (advisory), Champions-программу, mutation testing на критичных модулях, spec-driven workflow в paved-road-инструменте.
- **Что появляется.**
  - **Soundcheck (advisory)**: % сервисов в каталоге, % с актуальной AGENTS.md (правлен за 90 дней), % AI-PR с тегом `[ai]`, % T3 с PRR.
  - **Champions-программа**: 1 senior+ на Tribe; владеют локальной адаптацией шаблона; ежеквартальная Guild-Champions встреча.
  - **Mutation testing** для критичных модулей T3 (mutmut / Stryker / PIT) — на main, ночными прогонами; fail-blocking только на T3-релизе.
  - **Threat model**: basic для всех T2, full для всех T3.
  - **Spec-driven workflow** в paved-road-инструменте (Fission-AI OpenSpec / GitHub Spec Kit / Kiro Specs) — ≥10 archived `openspec/changes/` к концу фазы.
- **Роли.** Те же + Champions активно работают на Tribe-уровне.
- **Gate-критерий выхода.** Soundcheck показывает данные по 100% сервисов; ≥1 Champion на каждом Tribe; ≥80% T2/T3 с CLAUDE.md, правленным за 90 дней; ≥10 archived openspec-changes.
- **KPI фазы.**
  - Главный KPI замерен повторно (vs baseline Фазы 1).
  - Время от идеи до зарегистрированного прототипа: индикативная цель -30…-50% от baseline (не SLA).
  - Доля Tribe-ов с активным Champion ≥80%.
- **Главные риски.**
  - *Soundcheck → hard gate раньше времени.* Команды бунтуют или начинают gaming (правят CLAUDE.md без сути). Контрмера: hard gate только при ≥95% advisory **без gaming-сигналов** — для этого нужен периодический случайный аудит, иначе Goodhart.
  - *Champions становятся «AI police».* Контрмера: хартия Champion-а — устранять трение, не штрафовать; нарушения эскалируются в Guild, не Champion решает.
  - *Mutation testing замедляет CI.* Контрмера: только критичные модули (≤10% кода), ночные прогоны, fail-blocking только на T3-релизе.

### 7.5. Фаза 3. Hard gates + DORA + sunset (6–12 месяцев)

- **Цель.** Soundcheck advisory → hard gate. Ежеквартальные DORA-замеры отдельно по AI-PR. Sunset-кампании по zombie. Устоявшийся ритм.
- **Что появляется.**
  - **Hard gate**: PR не мержится без CLAUDE.md в репо; без `[ai]`-tag при детектированном AI-паттерне (warn в Фазе 2; fail в Фазе 3 после стабильного adoption за ≥3 квартала подряд, иначе остаётся warn); без OpenSpec proposal для T2+ из скоупа §6 библии.
  - **Ежеквартальные DORA-замеры** (deploy frequency, lead time, change failure rate, MTTR) — отдельно для AI-помеченных PR vs human-only.
  - **Sunset-кампании**: квартальная «охота на зомби» — все T0/T1 без коммитов >180 дней архивируются автоматически.
  - **Guild-ритм**: ежеквартальный пересмотр библии (по reasoned ADR в ядре §§1–7 + §4½; Appendix — свободнее); ежеквартальная встреча Guild ↔ Champions.
- **Роли.** Устоявшиеся; Guild — постоянный орган с календарём.
- **Gate-критерий выхода.** Фаза постоянна. «Выход» = установившийся квартальный ритм без авральных правок.
- **KPI фазы.**
  - Главный KPI: время от идеи до прототипа стабильно низкое; доля прототипов T1→T3 за год измерима и публикуется.
  - DORA по AI-PR: change failure rate ≤ DORA-elite, lead time ≤ DORA-high.
  - 0 zombie-репо в каталоге (по факту последнего sunset-цикла).
- **Главные риски.**
  - *DORA-метрики gamed.* Теги ставятся только на «безопасные» AI-PR. Контрмера: квартальный случайный аудит на 5–10 PR с детектированием AI-паттерна; при систематическом занижении — пересмотр Tribe-метрик и разговор Guild ↔ Tribe lead.
  - *Hard gate без paved road.* Большие батчи, обходы. Контрмера: gate-критерий из Фазы 2 (≥95% advisory без gaming) — обязателен; иначе остаёмся на advisory.
  - *Guild «забалтывается»* — 50 правок библии за квартал. Контрмера: ядро меняется только по reasoned ADR; Appendix-ы — свободнее; ритм правок ядра — не чаще раза в квартал.
