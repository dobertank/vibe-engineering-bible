# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Что это за репозиторий

Это **документационный репо** — политика по работе с AI-ассистентами для команд любого размера: от соло-разработчика и OSS-мейнтейнера до крупной организации. Кода, тестов, сборки, линтеров здесь нет. Не предлагай команды `npm`, `pytest`, `make` и т.п. — их некуда применять.

Содержит четыре файла на русском языке:

- **`vibe-engineering-bible.md`** — каноническая «библия». Универсальное ядро (§§1–10): два завета (Свободы для T0, Дисциплины для T1+), восемь заповедей, тринадцать грехов (§4½ — нормированный шорткодный язык отказа в мердже), DoD-таблица по T0–T3 с маркером `[ENT]` для опциональных корпоративных строк, spec-driven workflow, чек-лист ревьюера AI-кода, принципы baseline-шаблона, KPI масштабирования, глоссарий. Корпоративная специфика — в Appendix A (регламентированный T3-релиз: ИФТ/НТ/ЦКБ/PoB/«вывод в ПРОМ»), Appendix C (paved-road конкретной организации). Источники — Appendix D.
- **`CLAUDE.md.template.md`** — **универсальный** standalone-шаблон, дефолт. Без зависимости от Backstage, AI-Gateway, корп-Slack-каналов. Подходит соло-разработчику, OSS-команде, малой команде.
- **`CLAUDE.md.template.enterprise.md`** — **корпоративный** standalone-шаблон. Берётся, если у организации развёрнуты каталог сервисов, AI-Gateway, корп-Slack-каналы, политика обязательного PR-template.
- **`adoption-roadmap.md`** — универсальный roadmap внедрения подхода библии для трёх профилей: малая (1–10), средняя (10–100), крупная (100+). Поглотил Appendix B библии. Содержит фазы по неделям/месяцам с gate-критериями, KPI на фазу, anti-patterns внедрения, риски с контрмерами. Команды читают только свой профиль.

Шаблоны — единственный источник истины для baseline. Библия в §8 описывает принципы и упоминает оба варианта; тело шаблонов живёт в одном месте — в соответствующих файлах `CLAUDE.md.template*.md`.

## Семантическая связь файлов

- **Библия** держит общую модель: заповеди, грехи, DoD, KPI, spec-driven, чек-лист ревьюера.
- **Шаблоны** держат конкретные инструкции для AI-агентов в продуктовых репо. Команды копируют один из вариантов как `CLAUDE.md` или `AGENTS.md`, дополняют только разделом 8 (репо-специфика); разделы 1–7 не редактируют.
- При правке библии или одного шаблона проверяй остальные файлы на согласованность по смыслу. Маппинг не 1:1:
  - **Заповеди §4 библии → §4 шаблонов:** V/VI → §4.1–4.2 (безопасность, зависимости); VII → §4.3–4.4 (размер, тесты); VIII → §4.6 (документация). При правке заповеди — проверь обе версии шаблона.
  - **Заповедь IV (один стек, paved road) → §2 шаблонов** («Стек и команды») и Appendix C библии. Конкретный список paved-road-языков организации — только в Appendix C и/или в §2 enterprise-шаблона; универсальный шаблон содержит плейсхолдер.
  - **DoD-таблица §5 библии → §6 шаблонов** («Definition of Done для этой зоны»). В §6 шаблонов предзаполнены три варианта (T1/T2/T3); команда удаляет два неприменимых. Строки `[ENT]` есть только в enterprise-шаблоне; в универсальном они опущены и заменены родовыми эквивалентами (CI/CD, runbook). Правишь строку DoD — проверь оба шаблона + соответствие Appendix A (там расшифровка `[ENT]`-строк).
  - **Spec-driven §6 библии → §5 шаблонов** (блок «Для T2+ изменений…») **и §7 библии п.8** (чек-лист ревьюера). Любая правка артефактов `openspec/changes/<x>/{proposal,specs,design,tasks}.md` должна синхронно отразиться во всех трёх местах.
  - **Заветы §2–§3 библии** не имеют прямого аналога в шаблонах: T0 в шаблонах не описывается (шаблон ставится в T1+ репо). Завет Свободы — для людей, не для AI-агентов в продуктовых репо.
  - **Грехи §4½ библии** — нормированный словарь шорткодов отказа в мердже. В шаблоне прямого аналога нет (шаблон описывает правила, грехи — язык их нарушения). При добавлении/удалении греха — синхронизуй перекрёстные ссылки `*Грехи здесь:*` в соответствующей заповеди §4 и записи в глоссарии §10. Каждая строка таблицы §4½ обязана иметь действующий «корень» в §4/§5/§7 — грех без корня писать нельзя.
  - **§9 библии «Масштабирование» → `adoption-roadmap.md`.** §9 — общая рамка (главный KPI, под размер команды, paved road > мандат). Конкретный план внедрения для всех трёх профилей — в отдельном документе. Правишь фазу/KPI/риск в roadmap — проверь §9 на согласованность; правишь §9 — проверь, не появились ли расхождения с фазами в roadmap.
  - **Appendix A библии (роли T3-релиза) → Профиль III roadmap.** Delivery Manager / PO / QA / ЦКБ / Эксперт сопровождения остаются в Appendix A; план внедрения только ссылается, что профиль III применим, если такой процесс есть. Дублировать роли в roadmap не надо.
  - **Источники цифр §1 и §4½ → Appendix D библии**. Любое утверждение с цифрой или CVE должно иметь ссылку в Appendix D. Менять цифры — только при появлении новой версии исследования и с обновлением ссылки.

## Принципы при редактировании этих файлов

Документ сам себя описывает — соблюдай его правила, когда вносишь правки:

- **Краткость важнее полноты.** Цель корневого `CLAUDE.md` в продуктовом репо — ≤2000 токенов. Если строку можно убрать без потери смысла — убери.
- **Императив, а не описание.** «Используй pytest» — да. «Этот проект исторически использует pytest» — нет.
- **Конкретика, а не идеология.** Правила должны быть проверяемыми.
- Библия структурирована по разделам с номерами (1–10) плюс §4½ «Грехи» между Заповедями (§4) и DoD (§5), плюс Appendix A–D после §10. Дробный номер §4½ — намеренный, чтобы не ломать якорные ссылки на §5–§10. При вставке нового материала сохраняй эту схему: новые разделы — только как 4½/4¾, внутри существующих или как новый Appendix.
- Пиши и отвечай **по-русски** — все файлы и весь дискурс на русском.
- Коммиты — Conventional Commits (`feat`, `fix`, `chore`, `docs`, ...).

## Что НЕ делать

- Не превращай эти документы в README с историей проекта или маркетингом.
- Не возвращай дубликат шаблонов в библию: §8 библии содержит только принципы и ссылки на оба шаблона, тело шаблонов живёт в `CLAUDE.md.template.md` и `CLAUDE.md.template.enterprise.md`.
- Не добавляй в шаблоны репо-специфику — для неё в шаблоне есть раздел 8 (его заполняет команда-владелец сервисного репо, не этот репо).
- Не вставляй ссылки/цитаты, которых нет в Appendix D, без явного запроса пользователя — Appendix D намеренно опирается на проверенные публикации (METR, USENIX Spracklen, GitClear, Veracode, Stanford Perry, CodeRabbit, Apiiro, Mollick/BCG, CVE-перечень MCP, Anthropic auto-mode и т.п.).
- Не редактируй цифры в §1 и §4½ «на глаз». Проценты, выборки, годы, CVE-номера, CVSS пиннятся к конкретным публикациям из Appendix D. Менять — только при появлении новой версии исследования и с обновлением ссылки в Appendix D в том же коммите.
- Не вноси корпоративную пристёжку обратно в ядро (§§1–9). Любое упоминание ИФТ/НТ/ЦКБ/PoB/«вывод в ПРОМ»/Backstage/Soundcheck/AI-Code Guild как обязательного — только в Appendix A/C или в `adoption-roadmap.md` для крупного профиля; в ядре они либо за маркером `[ENT]`, либо как опциональный пример.
- Не дублируй универсальный и enterprise шаблоны в полностью разные документы. Они совпадают по §§1–7 на уровне структуры; различия — точечные (Backstage-ссылка, AI-Gateway, корп-Slack, обязательный PR-template, `[ENT]`-строки в DoD §6, заголовок «корпоративные стандарты» против «базовые»). При синхронизации правок проходись по обоим.

## Терминология (минимум для навигации)

- **Завет Свободы / Завет Дисциплины** — мета-структура библии. Свобода — §2 для T0; Дисциплина — §3+ для T1+. Граница проходит по личному эксперименту.
- **T0 / T1 / T2 / T3** — четыре уровня кода (эксперимент / прототип / внутренний инструмент / продакшен). Каждому — строка в DoD §5 библии. Маппинг с lifecycle-метаданными (experiment/prototype/mvp/beta/production/deprecated) — в §3 библии.
- **Lethal trifecta** — (приватные данные) + (недоверенный вход) + (внешний канал). Формулировка Simon Willison.
- **Paved road / Golden Path** — поддерживаемый путь по умолчанию.
- **Slopsquatting** — атака supply-chain через имена, которые часто галлюцинирует LLM.
- **`[ENT]`-маркер в DoD §5** — строка специфична для крупной организации с регламентированным релизом T3 (расшифровка — в Appendix A библии). В универсальном шаблоне `[ENT]`-строки опущены.
- **Релизный поток T3 (Appendix A)**: релизный тикет → дистрибутив → PoB → ИФТ → НТ → 4 параллельных заключения (PO, ЦКБ, SAST, сопровождение) → вывод в ПРОМ. Любой отказ из четырёх — откат. Применим только в крупных организациях.
- **Spec-driven workflow / OpenSpec** — для T2+: артефакты `openspec/changes/<x>/{proposal.md, specs/, design.md, tasks.md}`, после релиза delta мёрджится в `openspec/specs/`. §6 библии. OpenSpec поддерживает 20+ AI-агентов; конкретный paved-road-инструмент организации — в Appendix C.
- **Грехи (mortal / venial)** — нормированный словарь из 13 шорткодов (§4½ библии). Смертные (7): `#sandbox-bypass`, `#trifecta`, `#promptable`, `#test-del`, `#merge-pray`, `#workslop`, `#hidden-ai`. Простительные (6): `#slot-machine`, `#self-auto`, `#confident-wrong`, `#tautological`, `#sycophancy`, `#zombie-t0`. Список нормированный: ревьюер не изобретает имена.
- **Корпоративные роли (Appendix A / Профиль III roadmap):** Delivery Manager, Product Owner, QA, ответственный за НТ, ЦКБ, проверки безопасности, эксперт сопровождения, CISO, AI-Code Guild. Применимы в крупной орг; в малой команде — owner-разработчик / тимлид / security-владелец совмещают.
