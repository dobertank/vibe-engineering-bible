# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Что это за репозиторий

Это **документационный репо** — корпоративная политика по работе с AI-ассистентами для блока в 2000+ человек. Кода, тестов, сборки, линтеров здесь нет. Не предлагай команды `npm`, `pytest`, `make` и т.п. — их некуда применять.

Содержит два файла на русском языке:

- **`vibe-engineering-bible.md`** — каноническая «библия»: два завета (Свободы для T0, Дисциплины для T1+), восемь заповедей, DoD-таблица по T0–T3, spec-driven workflow, чек-лист ревьюера AI-кода, принципы baseline-шаблона, план внедрения и роли релиза, глоссарий.
- **`CLAUDE.md.template.md`** — standalone-шаблон, который команды копируют в свои сервисные репо. **Это единственный источник истины для baseline.** Библия в §8 ссылается на этот файл, а не дублирует его содержимое.

## Семантическая связь файлов

- **Библия** держит общую модель: заповеди, DoD, роли, KPI.
- **Шаблон** держит конкретные инструкции для AI-агентов в продуктовых репо. Команды копируют его как `CLAUDE.md` или `AGENTS.md`, дополняют только разделом 8 (репо-специфика); разделы 1–7 не редактируют.
- При правке любого из двух файлов проверяй второй на согласованность по смыслу. Маппинг не 1:1:
  - **Заповеди §4 библии → §4 шаблона:** V/VI → §4.1–4.2 (безопасность, зависимости); VII → §4.3–4.4 (размер, тесты); VIII → §4.6 (документация).
  - **Заповедь IV (один стек, paved road) → §2 шаблона** («Стек и команды»). Добавление/удаление языка — обязательно оба места.
  - **DoD-таблица §5 библии → §6 шаблона** («Definition of Done для этой зоны»). Команды копируют строки таблицы по своей зоне T1/T2/T3. Правишь строку DoD — проверь, что §6 шаблона не противоречит формулировкам и колонке «Кто проверяет».
  - **Spec-driven §6 библии → §5 шаблона** (блок «Для T2+ изменений…») **и §7 библии п.8** (чек-лист ревьюера). Любая правка артефактов `openspec/changes/<x>/{proposal,specs,design,tasks}.md` должна синхронно отразиться во всех трёх местах.
  - **Заветы §2–§3 библии** не имеют прямого аналога в шаблоне: T0 в шаблоне не описывается (шаблон ставится в T1+ репо). Завет Свободы — для людей, не для AI-агентов в продуктовых репо.

## Принципы при редактировании этих файлов

Документ сам себя описывает — соблюдай его правила, когда вносишь правки:

- **Краткость важнее полноты.** Цель корневого `CLAUDE.md` в продуктовом репо — ≤2000 токенов. Если строку можно убрать без потери смысла — убери.
- **Императив, а не описание.** «Используй pytest» — да. «Этот проект исторически использует pytest» — нет.
- **Конкретика, а не идеология.** Правила должны быть проверяемыми.
- Библия структурирована по разделам с номерами (1–10). При вставке нового материала сохраняй сквозную нумерацию и не ломай якорные ссылки внутри документа.
- Пиши и отвечай **по-русски** — оба файла и весь дискурс на русском.
- Коммиты — Conventional Commits (`feat`, `fix`, `chore`, `docs`, ...).

## Что НЕ делать

- Не превращай эти документы в README с историей проекта или маркетингом.
- Не возвращай дубликат шаблона в библию: §8 библии содержит только принципы и ссылку на шаблон, тело шаблона живёт в одном месте — `CLAUDE.md.template.md`.
- Не добавляй в шаблон репо-специфику — для неё в шаблоне есть раздел 8 (его заполняет команда-владелец сервисного репо, не этот репо).
- Не вставляй ссылки/цитаты, которых нет в библии, без явного запроса пользователя — библия намеренно опирается на проверенные публикации (METR, USENIX Spracklen, GitClear, Veracode, Stanford Perry, CodeRabbit, Apiiro и т.п.).
- Не редактируй цифры в §1 библии «на глаз». Проценты, выборки, годы пиннятся к конкретным публикациям из этого же раздела. Менять — только при появлении новой версии исследования и с указанием источника в коммите.

## Терминология (минимум для навигации)

- **Завет Свободы / Завет Дисциплины** — мета-структура библии. Свобода — §2 для T0; Дисциплина — §3+ для T1+. Граница проходит по личному эксперименту: пока код не вышел за пределы твоей машины, действует Свобода с тремя нерушимыми границами; всё остальное живёт по Дисциплине.
- **T0 / T1 / T2 / T3** — четыре уровня кода (эксперимент / прототип / внутренний инструмент / продакшен). Каждому соответствует строка в DoD-таблице §5 библии.
- **Lethal trifecta** — (приватные данные) + (недоверенный вход) + (внешний канал). Агент не должен иметь все три одновременно. Формулировка Simon Willison.
- **Paved road / Golden Path** — поддерживаемый путь по умолчанию (Spotify, Netflix). Сойти можно через ADR.
- **Slopsquatting** — атака supply-chain через имена, которые часто галлюцинирует LLM.
- **Релизный поток T3** (из регламента): релизный тикет в Jira → дистрибутив → PoB → ИФТ → НТ → 4 параллельных заключения (PO, ЦКБ, SAST/безопасность, сопровождение) → вывод в ПРОМ. Любой отказ из четырёх — откат.
- **Spec-driven workflow / OpenSpec** — для T2+ обязателен договор о спецификации до кода: артефакты `openspec/changes/<x>/{proposal.md, specs/, design.md, tasks.md}`, после релиза delta мёрджится в `openspec/specs/`. Подробнее — §6 библии.
- **Роли:** Delivery Manager (драйвер релиза), Product Owner, QA, ответственный за НТ, ЦКБ, проверки безопасности, эксперт сопровождения. Owner библии и шаблона — AI-Code Guild.
